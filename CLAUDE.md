# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

**MOLGW** (Many-body perturbation theory for atoms, molecules, and clusters) is a computational chemistry code implementing advanced electronic structure methods. The project is written primarily in Fortran 2003/2008 with C++ components for integral evaluation. It supports HF, DFT, GW, MP2, CI, TDDFT, BSE, and many other advanced methods.

**Main entry point:** `/src/molgw.f90` (the program)

## Build System

The project uses a Makefile-based build system with machine-specific configuration files.

### Key Files
- `Makefile` (root): Top-level build orchestration
- `src/Makefile`: Main compilation rules with explicit module dependency levels (level000 through level999)
- `my_machine.arch`: Machine-specific configuration file (examples in `config/` directory)

### Build Commands

```bash
# Build the main executable
make

# Build with parallel jobs (recommended)
make -j 4

# Clean build artifacts
make clean

# Install to PREFIX (default: current directory)
make install PREFIX=/usr/local
```

### Machine Configuration

MOLGW requires setting compiler and library paths in `my_machine.arch`. Key variables:
- `FC`: Fortran compiler (gfortran, ifort, ifx, flang)
- `CXX`: C++ compiler
- `FCFLAGS`: Fortran compilation flags (e.g., `-O2 -cpp`)
- `CXXFLAGS`: C++ compilation flags
- `OPENMP`: OpenMP flag (leave empty for sequential)
- `LAPACK`: BLAS/LAPACK libraries (required)
- `LIBXC`: DFT functional library (required for DFT; leave empty for HF-only)
- `LIBXC_INCS`: LIBXC include path
- `LIBXC_ROOT`: Alternative to setting LIBXC directly
- `LIBCINT`: Gaussian integral library (required; use LIBCINT_ROOT to specify location)
- `LIBINT`: Alternative integral library (LIBCINT preferred)
- `SCALAPACK`: For distributed memory parallelization
- `FFTW3`, `HDF5`: Optional libraries for advanced features

Example configurations are in `config/my_machine_*.arch` (gfortran, Intel ifort/ifx, flang).

### Build Troubleshooting

The `src/Makefile` uses explicit module dependency levels because Fortran lacks automatic dependency resolution:
- **level000**: Foundational modules with no dependencies
- **level005-level999**: Progressive dependency chains
- If compilation fails due to missing modules, check that the source file containing the required module is in a lower level
- Parallel builds (`make -j`) require correct level ordering; sequential builds (`make`) will auto-retry

### Special Generated Files

These files are auto-generated at each build and should never be edited:
- `src/basis_path.f90`: Generated by `src/prepare_sourcecode.py`
- `src/revision.f90`: Generated by `src/prepare_sourcecode.py` (git SHA)
- `src/input_variables.f90`: Generated by `utils/input_variables.py` from `src/input_variables.yaml`

To add a new input variable:
1. Edit `src/input_variables.yaml`
2. Run `python3 utils/input_variables.py` to regenerate `input_variables.f90` and documentation

## Testing

The test suite is Python-based and flexible, supporting sequential and parallel execution.

### Run All Tests

```bash
# Sequential (1 thread, 1 core per test)
make test

# Parallel (4 threads, 1 core per test)
make test NC=4

# Distributed MPI tests (4 MPI ranks, 1 core per test)
make test NP=4

# Distributed with custom MPI runner
make test NP=4 MPIRUN="mpirun -np"
```

### Run Specific Tests

```bash
cd tests
# Run a single test (filename, with or without path prefix / .in extension)
python3 run_testsuite.py --input be_qsgw.in --keep

# Run multiple specific tests
python3 run_testsuite.py --input be_qsgw.in ch2o_bse.in --keep

# Exclude specific tests
python3 run_testsuite.py --exclude be_qsgw.in

# Filter by input keyword value
python3 run_testsuite.py --input-parameter scf = 'LDA'

# List all available test names
python3 run_testsuite.py --list
```

### Test Structure

- `tests/run_testsuite.py`: Main test runner (handles input validation, execution, output checking)
- `tests/inputs/testsuite`: Database of all tests — defines which input files to run, their descriptions, per-test flags (`restart`, `noparallel`, `need_scalapack`, `need_gradients`, `need_libcint`, etc.), and the expected output checks (key string, reference value, field position, tolerance). **Edit this file when adding a new test.**
- `tests/inputs/`: ~100 test input files covering HF, DFT, GW, MP2, CI, TDDFT, BSE, NOFT, etc.
- `tests/data_for_tests/`: Reference data and expected outputs

Test runner supports:
- `--verbose` (`-v`): Display output for any failed test
- `--keep` (`-k`): Retain working directories (useful for debugging)
- `--nc N`: Use N OpenMP threads per test
- `--np N`: Use N MPI processes per test
- `--mpirun LAUNCHER`: Custom MPI launcher command
- Tests requiring ScaLAPACK, LIBCINT, FFTW3, etc. are automatically skipped if the binary was not compiled with those libraries

### Continuous Integration

GitHub workflows in `.github/workflows/` run tests on:
- gfortran (sequential, OpenMP, ScaLAPACK)
- flang (sequential)
- Tests automatically download and build libcint 6.1.2

## Code Architecture

### High-Level Flow (in `src/molgw.f90`)

1. **Initialization**
   - Read input file and echo configuration
   - Initialize atoms, basis sets, auxiliary basis (for RI)
   - Read/allocate molecular geometry and ECP pseudopotentials
   - Build one- and two-body Hamiltonians

2. **SCF Cycle** (`src/m_scf_loop.f90`)
   - Evaluate Fock/KS matrix
   - Diagonalize to get MO energies and coefficients
   - Build density matrix
   - Iterate until convergence (with LBFGS acceleration for geometry optimization)

3. **Post-Processing** (GW, MP2, CI, TDDFT, BSE, NOFT, etc.)
   - Compute correlation energies and/or spectral properties
   - Can be chained (e.g., GW@DFT, MP2@HF)

### Key Module Categories

**Quantum Chemistry Core**
- `m_hamiltonian_onebody.f90`: Kinetic energy, nuclear attraction, external fields
- `m_hamiltonian_twobodies.f90`: Two-electron integrals
- `m_dft_grid.f90`: DFT grid generation (Lebedev, Becke weighting)
- `m_eri.f90`: Electron repulsion integral (ERI) management
- `m_eri_calculate.f90`: ERI computation via LIBINT/LIBCINT
- `m_basis_set.f90`: Gaussian basis set handling

**Self-Energy & GW Methods**
- `m_gw_selfenergy_*.f90`: GW self-energy evaluation (analytic continuation, imaginary axis grids)
- `m_spectral_function.f90`: Spectral function (W matrix) for polarizability

**Density Functional Theory (DFT)**
- `m_libxc_tools.f90`: Bindings to LIBXC functional library
- `m_density_tools.f90`: Density matrix operations

**Many-Body Methods**
- `m_ci.f90`: Configuration interaction (CI) with Davidson diagonalization
- `m_mp2_energy.f90`: MP2 correlation energy
- `m_acfd.f90`: Adiabatic connection fluctuation-dissipation (ACFD)
- `m_build_bse.f90`: Bethe-Salpeter equation (BSE) kernel and solver
- `m_linear_response.f90`: Linear-response TDDFT/TDHF
- `m_realtime_tddft.f90`: Real-time propagation in TDDFT
- `m_dm_mbpt.f90`: Density-matrix perturbation theory (PT2/PT3, QSGW, QSMP2)

**NOFT (Natural Orbital Functional Theory)**
- `src/noft/` subdirectory: Orbital optimization, density matrix methods
- `m_noft_driver.F90`: NOFT main loop
- `m_optocc.F90`, `m_optorb.F90`: Occupation and orbital optimization

**Infrastructure**
- `m_inputparam.f90`: Input file parsing
- `m_io.f90`: File I/O (input/output files, restart files)
- `m_mpi.f90`: MPI wrapper for distributed memory (OpenMP via compiler flags)
- `m_scalapack.f90`: ScaLAPACK wrapper for distributed linear algebra
- `m_scf.f90`: SCF state variables (occupied/virtual orbitals, energies)
- `m_definitions.f90`: Global constants and kind parameters (dp = double precision)
- `m_warning.f90`, `m_timing.f90`, `m_memory.f90`: Diagnostic utilities

### Data Flow

Input file → parse `input_variables.f90` → initialize basis/atoms → build Ham → SCF → post-processing → output YAML/files

### Module Dependency Structure

Modules are organized in strict dependency levels in `src/Makefile` to ensure compilation order. Key principles:
- Level 0: No external dependencies (definitions, constants)
- Levels 1-5: Utility and infrastructure modules
- Levels 10+: Physics modules (basis, ERI, Hamiltonian)
- Levels 20+: SCF and advanced methods
- Levels 30+: Post-processing (GW, CI, BSE, etc.)

When adding a new module, place it at the appropriate level and update the Makefile.

## Language and Coding Conventions

### Languages
- **Fortran 2003/2008**: Main codebase (77 source files in `src/`, 16 in `src/noft/`)
- **C++**: LIBINT integral wrapper (4 files: `libint_onebody.cpp`, `libint_twobody.cpp`, and gradient variants)
- **Python**: Test suite, utilities, optional `molgw` Python package for automation

### Fortran Style
- Implicit none (required)
- Free-form source (F90 extension)
- Intent declarations (`in`/`out`/`inout`) on subroutine arguments (mandatory)
- Double-space indentation (2-space, not tabs)
- One-character variable names discouraged
- Use `private`/`protected` attributes to restrict module data access
- Avoid cascading object access (`a%b%c`); use methods instead
- MPI statements hidden in `m_mpi.f90`; use wrapper subroutines
- Variables declared in section after `implicit none`, then blank line, then executable code

### Naming Conventions
- Module names: `m_*.f90` (e.g., `m_basis_set.f90`)
- NOFT modules: Fortran fixed-form `.F90` files in `src/noft/`
- Test inputs: `*.in` files (e.g., `be_qsgw.in`)
- Loops: Use descriptive names (`istate`, `ibasis`, `ijob`) not single letters
- Constants/parameters: UPPERCASE
- Variables: lowercase
- Real/complex types: explicit via `real(dp)` or `complex(dp)` (kind parameter `dp` from `m_definitions`)

## Input and Output

### Input Files
- Simple format: `keyword = value`
- ~150 keywords supported (documented in `docs/input_variables.md`)
- Example inputs in `tests/inputs/` (e.g., `be_qsgw.in`, `ch2o_bse.in`)

### Output Files
- `molgw.yaml`: Structured output (energies, orbital info, transitions, etc.)
- `ENERGY_QP`: Quasiparticle energies (GW)
- `RESTART`: Binary wavefunction dump for restarting
- `SCREENED_COULOMB`: Polarizability data
- Log to stdout: Summary of input, SCF steps, timings

## Debugging and Development

### Helpful Compiler Flags
- `-O0 -g`: Debug symbols (slow)
- `-O2`: Fast, safe (default)
- `-fcheck=all` (gfortran): Runtime bounds checking
- `-traceback` (gfortran): Detailed error messages
- `-cpp`: Preprocess (required; already in most configs)

### Useful Modules for Debugging
- `m_warning.f90`: Print warnings/errors to stderr
- `m_timing.f90`: Measure section timing
- `m_memory.f90`: Track allocated memory
- `m_io.f90`: Safe I/O with error handling

## Key Files for Different Tasks

| Task | Key Files |
|------|-----------|
| Add new method | `src/m_*.f90` (new module) + update `src/Makefile` levels + call from `molgw.f90` |
| Add input keyword | `src/input_variables.yaml` → run `utils/input_variables.py` |
| Fix integral evaluation | `src/libint_*.cpp` or `src/m_eri_calculate.f90` |
| Debug SCF | `src/m_scf_loop.f90`, `src/m_hamiltonian_twobodies.f90` |
| Extend GW | `src/m_gw_selfenergy_*.f90`, `src/m_spectral_function.f90` |
| Add test | Create `tests/inputs/new_test.in`, add entry to `tests/inputs/testsuite` (input name, description, expected output checks) |
| Optimize compilation | Update `my_machine.arch` or `src/Makefile` (e.g., FCFLAGS, LIBXC_ROOT) |

## Common Workflows

### Build and Test (Quick Check)
```bash
cp config/my_machine_gfortran_seq.arch my_machine.arch
# Edit my_machine.arch to point to your libraries
make -j 4
make test NC=2
```

### Add a New Quantum Chemistry Method
1. Create `src/m_mynewmethod.f90` with module `m_mynewmethod`
2. Implement subroutines (follow existing code style)
3. Add to `src/Makefile` at appropriate dependency level
4. Call from `src/molgw.f90` main program
5. Add input variable(s) in `src/input_variables.yaml` if needed
6. Create test input file in `tests/inputs/`
7. Run `make -j 4 && make test`

### Modify Fortran and Recompile
```bash
cd src
make clean  # Optional; incremental build usually works
make -j 4
cd ..
# Executable is ./molgw
```

### Run a Single Calculation
```bash
./molgw tests/inputs/be_qsgw.in > be_qsgw.out
# Output: molgw.yaml, energy.out, etc.
```
